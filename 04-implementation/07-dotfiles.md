# Dotfiles and portable config

A short section, mostly to draw one line clearly: what's safe to keep
in a portable, git-tracked dotfiles repo, and what's inherently
per-machine and shouldn't be templated as if it were portable.

## Portable — belongs in a dotfiles repo

- `~/.ssh/config` — the `Host` blocks listing all 4 `IdentityFile`
  entries from `04-implementation/03-fido2-ssh-keys.md`. This file
  contains no secrets; it just lists which local stub filenames to try.
- `~/.gitconfig` — `user.signingkey`, `commit.gpgsign`, and general git
  preferences.
- Shell config (`~/.config/fish/config.fish` or equivalent) — aliases,
  prompt setup, environment variables unrelated to key material.
- This repo itself — the documentation you're reading.

## Not portable — inherently per-machine, don't template

- **The actual FIDO2 resident-key stub files** (`~/.ssh/id_yk_key1`
  etc.) generated by `ssh-keygen -K` on that specific machine. These are
  local pointers to hardware-resident credentials, not something to
  copy between machines — a fresh machine regenerates them directly
  from whichever card is plugged in, per `05-bootstrap/`.
- **LUKS enrollments** — which cards a given machine's disk accepts is
  a property of that machine's LUKS header, set once during
  `04-implementation/04-luks-disk-encryption.md`, not something
  expressed in a dotfile.
- **`/etc/pam.d/sudo` and `~/.config/Yubico/u2f_keys`** — similarly
  machine-specific; a NixOS follow-up document can make this
  declarative, but on Arch/Fedora/Debian these are per-machine state
  set once during `04-implementation/05-sudo-pam-u2f.md`.
- **GPG's local trust database** (`~/.gnupg/trustdb.gpg`) — rebuilt
  automatically after importing the public key and setting ultimate
  trust once per machine; not meaningfully portable as a file, and
  regenerating it is trivial.

## The practical shape of a new machine's setup

Given the split above, bringing a new machine online is really two
independent things happening in parallel: cloning the portable dotfiles
repo (which gives you SSH config and git config instantly), and running
the per-machine provisioning steps once (LUKS enrollment, sudo PAM,
pulling FIDO2 stubs and the GPG public key down fresh). Neither depends
on the other finishing first. Full sequencing in `05-bootstrap/`.
